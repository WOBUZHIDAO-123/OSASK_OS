# IDT（中断描述符表）和门描述符学习笔记

## 什么是中断？

中断就是 CPU 正在执行代码时，**突然被某件事打断**，需要暂停当前工作去处理那件事。

### 中断的两种来源

| 类型 | 来源 | 例子 |
|------|------|------|
| **异常（Exception）** | CPU 内部自己触发 | 除零错误、页错误、无效操作码 |
| **硬件中断（Interrupt）** | 外部设备通过 PIC 通知 CPU | 键盘按下、定时器到期、鼠标移动 |

### 中断的处理流程

```
CPU 正在执行 for(;;) io_hlt()
                    ↓
         键盘被按下 → PIC 通知 CPU
                    ↓
   CPU 暂停当前工作，保存现场
                    ↓
   CPU 去查 IDT[键盘中断号] → 拿到处理函数地址
                    ↓
   CPU 跳转到键盘处理函数执行
                    ↓
   处理完成 → 恢复现场 → 继续执行 for(;;) io_hlt()
```

## IDT（中断描述符表）是什么？

IDT 是一张**中断号码 → 处理函数的翻译表**，放在内存的 `0x0026f800` 处。

```
IDT 在内存中的布局：
0x0026f800 ┌──────────────────┐
           │ IDT[0]  门描述符  │ ← 除零错误
           ├──────────────────┤
           │ IDT[1]  门描述符  │ ← 调试异常
           ├──────────────────┤
           │ ...              │
           ├──────────────────┤
           │ IDT[0x21] 门描述符│ ← 键盘中断（IRQ1）
           ├──────────────────┤
           │ ...              │
           ├──────────────────┤
           │ IDT[255] 门描述符 │
0x0026FFFF └──────────────────┘

每个门描述符 = 8 字节
总共 256 个 × 8 字节 = 2KB
```

---

## 门描述符（GATE_DESCRIPTOR）结构

每个门描述符 **8 字节**，定义了一个中断号的"处理方案"：

```c
struct GATE_DESCRIPTOR {
    short offset_low;     // [0-1]  处理函数地址低 16 位
    short selector;       // [2-3]  目标代码段选择子（用哪个段来执行）
    char  dw_count;       // [4]    调用门参数计数（中断门填 0）
    char  access_right;   // [5]    访问权（P、DPL、TYPE）
    short offset_high;    // [6-7]  处理函数地址高 16 位
};
```

### 内存布局图

```
偏移  字节     内容            说明
────────────────────────────────────────────
+0    [0-1]   offset_low     处理函数地址的 bit0~bit15
+2    [2-3]   selector       代码段选择子（通常 = 2*8 = 16）
+4    [4]     dw_count       调用门专用参数计数（中断门=0）
+5    [5]     access_right   权限字节
+6    [6-7]   offset_high    处理函数地址的 bit16~bit31

最终处理函数地址 = offset_high << 16 | offset_low
                 = 32 位物理地址
```

### 画个直观的图

```
门描述符（8 字节）:
┌───────┬───────┬────┬────┬───────┐
│offset │selector│dw  │acc │offset │
│_low   │       │cnt │right│_high  │
│16bit  │16bit  │8bit│8bit│16bit  │
└───┬───┴───────┴────┴────┴───┬───┘
    │                          │
    └────────┬─────────────────┘
             ↓
    处理函数 32 位地址
```

## access_right 权限字节

第 5 字节的 access_right 控制这个中断的权限：

```
bit7  : P（存在位）— 1=有效，0=无效
bit6-5: DPL（描述符特权级）— 0=内核级，3=用户级
bit4  : 固定为 0（系统段标志）
bit3-0: TYPE — 门的类型
```

### 常用的 TYPE 值

| TYPE | 名称 | 作用 |
|------|------|------|
| `0x0E`（= 0xE） | **中断门** | IF 自动清除，防止中断嵌套 |
| `0x0F`（= 0xF） | **陷阱门** | IF 不清除，用于调试 |
| `0x05`（= 0x5） | **任务门** | 任务切换专用 |

### 为什么中断门要自动清 IF？

```
CPU 正在处理键盘中断（IF=0）
                    ↓
      突然又来了一个定时器中断
                    ↓
   如果不清 IF，CPU 会嵌套处理另一个中断
   可能导致栈溢出或数据不一致
                    ↓
中断门自动清 IF = "我正在忙，别打断我"
陷阱门不清 IF = "可以打断我，方便单步调试"
```

### 常见 ar 值

| ar | 二进制 | 含义 |
|----|--------|------|
| `0x8E` | `1000 1110` | P=1, DPL=0, TYPE=0xE → **内核级中断门** |
| `0xEE` | `1110 1110` | P=1, DPL=3, TYPE=0xE → **用户级中断门** |

## set_gatedesc 函数拆解

```c
void set_gatedesc(struct GATE_DESCRIPTOR *gd, int offset, int selector, int ar)
{
    // 把 32 位的处理函数地址拆成两半
    gd->offset_low   = offset & 0xffff;          // 取低 16 位
    gd->offset_high  = (offset >> 16) & 0xffff;  // 取高 16 位

    // 设置代码段选择子（哪个 GDT 段来执行这个中断处理函数）
    gd->selector     = selector;                  // 通常 = 2*8 = 16

    // 调用门才用的参数计数
    gd->dw_count     = (ar >> 8) & 0xff;          // 中断门 = 0

    // 权限字节
    gd->access_right = ar & 0xff;                 // P + DPL + TYPE
}
```

### 地址拆分示意

```
offset = 0x56781234（假设的处理函数地址）

高位 16 位 → 0x5678    低位 16 位 → 0x1234

offset_high = 0x5678    offset_low = 0x1234

拼回去：0x5678 << 16 | 0x1234 = 0x56781234 ✅
```

## IDT 初始化（当前第 6 天的状态）

```c
// 全部清零，256 个中断都没注册处理函数
for (i = 0; i < 256; i++) {
    set_gatedesc(idt + i, 0, 0, 0);
}
load_idtr(0x7ff, 0x0026f800);  // 告诉 CPU IDT 在 0x26f800
```

**所有 256 个门描述符**：

```
offset_low   = 0
selector     = 0
dw_count     = 0
access_right = 0  → P=0（不存在）
offset_high  = 0

→ 整个门描述符无效，CPU 不许用
```

## 等第 10 天以后注册键盘中断时

```c
// 假设键盘中断处理函数叫 keyboard_handler
// 它的地址 0x56781234，用 GDT[2]（选择子=16）执行
set_gatedesc(idt + 0x21, 0x56781234, 2*8, 0x8E);
```

填写后的 IDT[0x21]：

```
offset_low   = 0x1234
selector     = 16（GDT[2] = 代码段）
dw_count     = 0
access_right = 0x8E（P=1, DPL=0, TYPE=0xE = 中断门）
offset_high  = 0x5678

CPU 解析：
处理函数地址 = 0x5678 << 16 | 0x1234 = 0x56781234 ✅
权限检查 = 中断门，存在，内核级
```

这样键盘一按，CPU 就能找到正确的处理函数去执行。

## GDT vs IDT 总结

| | GDT | IDT |
|----|-----|-----|
| 全称 | 全局描述符表 | 中断描述符表 |
| 位置 | `0x00270000` | `0x0026f800` |
| 大小 | 64KB（8192 个） | 2KB（256 个） |
| 描述符 | 段描述符（8 字节） | 门描述符（8 字节） |
| 描述符内容 | 基址 + 界限 + 权限 | 处理函数地址 + 选择子 + 权限 |
| 作用 | **定义内存段的权限** | **定义中断来了该去哪执行** |
| 类比 | 门禁权限表 | 紧急电话簿 |

### 一个中断的完整处理链路

```
设备触发中断
    ↓
PIC 通知 CPU（中断号 = 0x21）
    ↓
CPU 查 IDT[0x21] → 拿到门描述符
    ↓
门描述符里写着：
  - 处理函数地址 = 0x56781234
  - 用 GDT[2] 来执行（代码段）
    ↓
CPU 检查 GDT[2] 的权限 → 可执行 ✅
    ↓
CPU 跳转到 0x56781234 处理键盘事件
```
